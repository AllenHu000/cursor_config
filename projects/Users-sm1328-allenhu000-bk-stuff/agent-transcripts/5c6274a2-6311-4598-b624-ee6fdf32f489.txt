- **编辑**：用 `orderId` 是否已等于当前值来判断是否跳过（语义是「已加载过该订单」）。
- **新增**：用布尔 `apiRequestLocked`，且**没有 try/finally**，接口抛错时锁不会释放。

**现在的做法：**

- **同一套锁**：用唯一锁 key 表示「当前正在执行的任务」：
  - 编辑：`edit_${orderId}`
  - 新增：`add`
- **入口统一**：先算 `lockKey`，若 `max.data.get(PROP_CHANGE_LOCK_KEY) === lockKey` 直接 return，避免同一任务并发。
- **编辑保留原有语义**：若当前已加载的订单就是本次 `orderId`，也直接跳过（避免重复拉同一单）。
- **try/finally 保证释放**：无论成功、失败都会在 `finally` 里清空锁，避免异常导致锁常驻。

这样编辑/新增都用「同一个锁 key + 同一套加锁/解锁逻辑」，差异只在 key 的生成方式，更一致也更安全。

---

## 2. 已做的其它小优化

- **`queryCommonData` 抽到外层**：并改为显式传入 `(dispatch, CreateOrder)`，便于复用和单测；编辑/新增分支改为调用 `handleEdit` / `handleAdd`，主流程更清晰。
- **`selectedService` 空值保护**：`max.data.get('selectedService') || {}`，避免 `undefined` 时 `Object.keys` 报错。
- **`spaceResources?.[0]`**：避免 `spaceResources` 为空或缺失时取 `[0]` 报错。
- **移除调试用 `console.log`**：减少噪音；如需排查可改用带开关的 logger 或仅在开发环境打。
- **JSDoc**：为锁常量和 `getPropChangeLockKey` 补了注释，返回类型改为 `Promise<void>`。

## 3. 重叠组划分：`getOverlapGroups`

```419:443:src/MCalendarView/MCalendar.js
        static getOverlapGroups(orders) {
            if (!orders || orders.length === 0) return [];
            const assigned = new Set();
            const groups = [];

            for (const order of orders) {
                if (assigned.has(order)) continue;
                const group = [order];
                assigned.add(order);
                const stack = [order];
                while (stack.length) {
                    const current = stack.pop();
                    for (const other of orders) {
                        if (assigned.has(other)) continue;
                        if (this.isTimeOverlap(current, other)) {
                            assigned.add(other);
                            group.push(other);
                            stack.push(other);
                        }
                    }
                }
                groups.push(group);
            }
            return groups;
        }
```

作用：把**同一资源**下的订单，按「时间交叉」关系分成若干**重叠组**，且是**传递闭包**：

- 若 A 与 B 交叉、B 与 C 交叉，则 A、B、C 会在同一组（即使 A、C 不直接交叉）。
- 与组内任何订单都不交叉的订单，单独成组。

实现方式：对每个尚未归属的订单做一次「种子」BFS：用栈从该订单出发，把所有与当前订单交叉的、且未分配的订单加入同一组并入栈，直到没有新的交叉订单。这样得到多个 `group`，每个 `group` 是「互相（直接或间接）交叉」的订单数组。

意义：**只有同一组内的订单会一起均分这一行的高度**；不同组之间互不影响，所以「不相关订单」可以单独占满整行。

  - A 和 B 必须不同行（例如 A=line0, B=line1）；  
  - B 和 C 必须不同行（C 不能也是 line1，否则和 B 冲突）；  
  - A 和 C 可以同一行（都可以放在 line0）。

正确分配可以是：A=0, B=1, C=0（两行就够了）。  
如果不在 `assignLinesToOrders` 里对组内订单两两做 `isTimeOverlap`，就得不到这种「只让真正重叠的订单占不同行」的结果。

---

## 小结

| 步骤 | 作用 | 是否还需要检查「两两交叉」 |
|------|------|----------------------------|
| `getOverlapGroups` | 按「传递交叉」分组，决定哪些订单一起均分行高 | 不负责分配具体行号 |
| `assignLinesToOrders` | 在组内给每个订单一个 `line`，保证直接交叉的订单不同行 | **需要**，否则无法正确分配 line |

所以：**分组解决「和谁一起分高度」，行号分配解决「和谁不能在同一行」**；后者必须知道「谁和谁直接交叉」，因此组内仍然需要两层循环 + `isTimeOverlap`。
